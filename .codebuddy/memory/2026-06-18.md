## 优化启动数据库查询
- **GallerySyncService.FullSyncAsync** 消除 N+1 问题：替换 `Select(Gid)→HashSet + 逐个 FindAsync` 为单次 `ToDictionaryAsync`，500 个画廊从 500+ 次查询降到 1 次
- 清理已删除目录从两次额外查询（`Select(DirPath)` + `Where(IN)`)改为纯内存 `Except` 计算
- 批量保存从 50 改到 100 个目录一次
- **消除逐条 UPDATE SyncedAt**：`HasChanged()` 比较字段，未变化的不碰实体，最后用 `ExecuteUpdate` 批量更新 SyncedAt（1527 条→1 条 SQL）
- **Baseline 检测优化**：已迁移的数据库跳过 `manga` 表检查，从 3 条查询降到 2 条
- **GallerySync 进度日志降频**：每 100→500 条输出一次（19→4 条），SaveChanges 100→200
- 启动日志精简确认：DB 查询降至最低，文件扫描(~2s)是唯一时间瓶颈

## 修复 paged API 异常 + AlbumKey 重构
- **JsonArrayFirst/JsonArrayLength 不可翻译**：C# 方法在 EF LINQ 中无法转 SQL，改为 `StartsWith("[\"name\"")` → `LIKE` 和 `Contains(",")` → `LIKE`
- **AlbumKey 重构**：`LocalGallery` 新增 `AlbumKey` 字段 + 索引 + 迁移；GetPagedGalleries 用 `AlbumKey = key`/`AlbumKey IS NULL` 替代全部 `albumGids`/`excludeGids` 逻辑；AlbumsController.Save 同步更新；启动时一次性同步
- 前端不再需要传 `albumGids`，`albumOrder` 仅用于自定义排序
- 删除 `JsonArrayFirst`、`JsonArrayLength`、`ApplyMemSort` 死代码
